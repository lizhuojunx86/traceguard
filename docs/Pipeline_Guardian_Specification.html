System architecture

The 6-step self-optimization loop

Pipeline Guardian 的核心工作机制是一个闭环自优化系统：分析偏差 → 量化测试 → 提出改造 → 实施改造 → 回测验证 → 输出新工作流。当回测未达标时，系统会从 Step 5 循环回到 Step 2，直到达成目标或触发熔断。

Step 1

Analyze deviation risks in a workflow

偏差分析是整个 Pipeline Guardian 的"诊断"起点。它的工作机制分为两个阶段：

阶段一：静态解析

Guardian 读取你的 YAML 配置文件，提取每个步骤的输入输出合约——包括期望的 JSON Schema、必需字段、输出长度范围、目标语言等。这一步完全不需要运行 pipeline，纯粹从配置中推断出每个"接缝"处可能出错的方式。比如 market_02 的输出需要包含 leads 数组，但 LLM 有时候会输出一个纯文本段落而非 JSON——这就是一个"格式风险"。

阶段二：风险分类扫描

系统对每个步骤间的交接点扫描五类偏差风险：

  • 格式风险——输出不是合法 JSON、字段名拼写错误、嵌套结构不符合 schema。这是串联 pipeline 中最常见的崩溃原因。
  • 语义连贯风险——上一步给了美股数据，下一步却开始分析港股。输入和输出之间的语义断裂。
  • 输出漂移风险——同一个 prompt 在不同运行间产出结构不一致。第一天输出 5 个字段，第二天输出 3 个。这是 LLM 的固有不确定性。
  • 边界风险——输出被截断（达到 token 上限）或异常膨胀（LLM 进入重复循环），导致下游解析失败。
  • 语言风险——在中文 pipeline 中混入英文段落，或在多语言环境下语言标签不一致。

扫描完成后，系统为每个步骤生成一个风险图谱（Risk Map）：哪些风险类别存在、严重程度如何（高/中/低）、建议的校验策略是什么。这个 Risk Map 直接驱动下一步——系统根据风险等级决定为每个步骤配置什么样的 Guardian 检查规则，以及优先生成哪些测试用例来量化偏差。

Step 2

Quantify deviations with test cases

这一步是把"可能出问题"变成"用数据证明哪里出了问题、出了多少"。

测试用例从哪里来

历史回放是最有价值的来源。你的 pipeline 每天通过 cron 触发，每次运行都会被 Guardian 记录在 eval_store 里。积累一周后，系统就拥有了 7 组真实的输入输出对。回放时，系统把历史输入重新喂给每个步骤，观察输出是否与历史记录一致——如果同样的输入今天产出了不同结构的输出，这就是可量化的漂移。

合成边界用例由 LLM 根据 Risk Map 自动生成。如果 Step 1 识别出 market_02 有"截断风险"，系统会让 LLM 生成一个特别长的输入（逼近 token 上限），看这个步骤在极端条件下是否仍能产出合法输出。

黄金样本是你手动标注的"标准答案"。比如你知道一个理想的 market_03 市场报告应该长什么样，把它存为 golden sample，系统每次测试都会拿实际输出与它对比。这些样本数量不需要多，5-10 个足够建立质量锚点。

隔离执行器怎么工作

关键设计：不是跑整条 pipeline，而是逐步隔离运行。系统把测试输入直接注入某个步骤的容器，只运行这一个步骤，收集它的输出。这样做有两个好处：一是能精确定位问题出在哪一步（而不是只看到最终输出有问题）；二是节省成本——不需要每次都跑完整 pipeline。

同一个测试输入会被执行多次（默认 3 次），用来测量输出稳定性。如果三次运行产出的 JSON 结构完全一致，稳定性得分就是满分；如果第二次少了一个字段，稳定性就会下降。

三层评分的具体指标

结构层（零 LLM 开销）纯规则引擎，跑得最快：schema_pass 是输出通过 JSON Schema 校验的比例；field_complete 检查每个必需字段是否都存在且非空；length_ok 检测输出长度是否在历史分布的合理区间内（过短意味着截断，过长可能是重复循环）；lang_ok 用简单的字符集分析检测语言混用。如果你的 pipeline 有 10 组测试用例，其中 8 组通过 schema 校验，schema_pass 就是 80%。

语义层（轻量 LLM 调用）用 LLM-as-Judge 打 1-5 分。系统给评判模型一段极短的评估 prompt：这是步骤 X 的输入，这是它的输出，请从三个维度打分——输入输出相关性（上一步给了美股数据，这一步确实在分析美股吗）、内容质量（信息密度高不高，有没有明显的胡编）、逻辑连贯性（论述有没有自相矛盾）。每个维度 1-5 分，评判模型只需返回一个 JSON，token 消耗很小。

稳定层（多次运行对比）同一个输入跑 3 次，对比输出的变化。format_var 看三次输出的 JSON 结构是否一致（字段名、嵌套层级）；content_var 用简单的文本相似度（如 Jaccard 或 embedding cosine）衡量内容漂移程度；length_var 检查三次输出的长度波动是否在合理范围内。这一层直接捕捉"同一个 prompt 在不同运行间输出结构不一致"的问题。

偏差报告的核心产出

三层评分完成后，系统生成一份结构化的偏差报告，包含每个步骤的综合分数、最薄弱的指标、以及失败最严重的具体样本（附带原始输入和实际输出）。这份报告有两个用途：一是作为基线存入 eval_store，后续第 5 步回测时，系统会把优化后的分数与这个基线对比，用 delta 值量化改进幅度；二是直接驱动第 3 步的改造建议——如果 step_02 的 field_complete 只有 76%，系统知道应该优先修复这个步骤的输出约束。

整个量化过程对一条 4 步 pipeline、10 组测试用例、3 次重复运行来说，大约需要 120 次步骤执行 + 40 次 LLM 评判调用。按 MiniMax M2.7 的成本，总开销在几毛钱的级别。

Step 3

Generate improvement proposals

这是从"诊断"跨入"治疗"的关键一步——系统要回答的问题不再是"哪里有问题"，而是"怎么修"。

分诊：先修哪一步

引擎不会同时改所有步骤。它先对偏差报告中的步骤按综合严重度排序——结构分最低的排前面，因为结构失败会直接让下游崩溃；语义分低但结构通过的排后面，因为至少 pipeline 不会中断。用上一步的例子：step_02 analyze 的结构分只有 82%、语义分 68%、稳定性 72%，它会被排在第一优先级。

根因分类：为什么出错

这是整个提案引擎最关键的环节。系统把每个低分步骤的失败样本（来自 Step 2 存储的具体输入输出对）喂给 LLM，让它分析这些失败的共性原因，归入四个类别之一：

Prompt 问题——最常见，也最容易修。比如系统发现 step_02 的 20 个测试中有 5 个缺少 leads 字段，LLM 分析后发现原始 prompt 只说了"分析市场数据并输出结果"，根本没有指定输出必须包含哪些字段。这就是指令模糊导致的格式偏差。

合约问题——Guardian 的校验规则本身定义得太松。比如 schema 允许 leads 是字符串或数组，但下游步骤只能处理数组。这不是 LLM 的错，是接口合约没定义清楚。

配置问题——temperature 设太高导致输出不稳定，或 max_tokens 设太低导致截断。这类问题的根因分析相对简单，因为偏差报告中的稳定性得分和长度检查能直接指向。

架构问题——最严重但最少见。比如某个步骤被赋予了太多职责（同时做数据清洗、分析、和格式化），单个 LLM 调用难以稳定完成。系统会建议拆分步骤。

提案生成器的内部逻辑

引擎对每个提案的生成过程是这样的：它把失败样本（具体的输入输出对）、偏差报告中的量化指标、以及当前的 prompt / config 原文一起喂给 LLM，用一段结构化的元提示词（meta-prompt）要求它输出诊断和修复方案。元提示词大致是：

以下是 step_02 的当前 prompt、配置、以及 5 个失败测试样本。每个样本标注了具体的失败原因（字段缺失/语言混用/截断等）。请分析共性根因，提出具体修改方案，以 before/after diff 格式输出，并预估修改后各指标的改善幅度。

这里有一个关键设计——预测的影响值是估算而非承诺。系统标注的 "82% → 97%" 是 LLM 基于对失败模式的分析做出的合理推测，真实效果要等第 5 步回测才能验证。这就是为什么闭环设计至关重要。

为什么是"人在环"

每个提案底部都有 "Apply" 和 "Edit first" 两个按钮。这不是偷懒，而是深思熟虑的设计。在当前阶段，让 AI 自动改写工作流代码的风险太高。一个 prompt 改动看起来无害，但可能破坏下游三个步骤对它输出格式的依赖。所以系统只做到"生成提案"，由你来做最终决策。

被批准的提案才会进入第 4 步（实施改造），被拒绝的提案会被记录下来，作为 LLM 在未来生成提案时的负反馈：上次你拒绝了"降 temperature"的方案并附了理由"分析任务也需要一定创造性"，系统下次就不会再为同类步骤提出这种建议。

Step 4

Implement workflow changes

这一步是整个系统中最需要"谨慎"的环节。核心设计原则是：永远不直接改生产环境，所有改动先在沙箱里跑通。

第一道安全网：快照

在做任何改动之前，系统先把当前步骤的所有可变文件打一个快照，存为版本号 v(N)。这包括：prompt 文本文件、YAML 配置（temperature、max_tokens 等）、JSON Schema 定义、以及 Guardian 校验规则。快照存储在 eval_store 里，和 eval trace 数据放在一起。这意味着你随时可以一键回滚到任何历史版本——改坏了不怕，30 秒恢复。

改动分类：系统能改什么，不能改什么

这是整个 Step 4 最重要的设计边界。Pipeline Guardian 只操作四种类型的文件，全部是声明式配置，不涉及任何业务逻辑代码：

为什么只改配置，不改代码

这是一个深思熟虑的架构决策，不是能力限制。你的 agent 容器（market_01、market_02...）是独立的 Docker 镜像，内部可能有复杂的 Python 逻辑——数据清洗、API 调用、格式转换。如果 Guardian 去改写这些代码，一个变量名拼错就可能导致容器启动失败，而且排查成本极高。

但 prompt、temperature、schema 和 Guardian 规则这四样东西都是声明式配置——它们描述"要什么"而不是"怎么做"。改一个 prompt 文件不会让容器崩溃，最坏的情况是 LLM 输出变差，而这正是 Step 5 回测能捕获的。改一个 temperature 值更是零风险的操作。这种"只改旋钮，不改引擎"的策略，是让自动化改造可以安全执行的根本前提。

Out of scope — Guardian never touches these
Agent container code (Python, JS, etc.) — requires human developer
Docker Compose orchestration — container order, networking
External API credentials or endpoints
Database schemas or data stores

沙箱烟雾测试：改完先试跑

配置修改完成后，系统不会直接推送到生产环境。它先在沙箱中做一个快速验证：用 3 个测试用例（1 个历史正常案例 + 1 个历史失败案例 + 1 个边界案例）跑一遍被修改的步骤。沙箱检查三件事：

容器是否能正常启动并返回输出（没有因为 prompt 格式错误而报错）；输出是否通过了新版 schema 的结构校验（如果 schema 也被改了）；以及新输出的语义分是否不低于旧版本（防止"修了格式但内容变差"）。

如果三个测试中任何一个失败，系统自动回滚到 v(N) 快照，标记这个提案为"沙箱失败"，并把失败原因记录到 eval_store——下次提案引擎生成建议时会参考这条记录，避免重蹈覆辙。

版本推进：从沙箱到生产

沙箱通过后，系统将修改后的配置文件标记为 v(N+1) 并写入生产路径。整个推进过程会生成一条变更记录：

change_id:    chg_20260321_step02_001
version:      v6 → v7
timestamp:    2026-03-21T22:30:00+08:00
proposal_id:  prop_003
changes:
  - file: prompts/step_02.txt (v6 → v7)
  - file: configs/market_intel.yaml (temperature: 0.7 → 0.3)
sandbox:      PASS (3/3 cases)
status:       PROMOTED

这条记录是 Step 5 回测的起点——回测时，系统知道 v7 做了什么改动，可以精确比较 v6 和 v7 在完整测试集上的表现差异。

如果回测发现改坏了呢？

这正是为什么快照机制如此重要。如果 Step 5 的大规模回测显示 v7 的某些指标反而下降了，系统有两个选择：回退到 v6（安全的一键操作），或者进入下一轮迭代——Step 5 的失败数据会喂回 Step 3 的提案引擎，生成一个更好的修复方案。这就是整个 6 步闭环中"5 → 2 → 3 → 4 → 5"的迭代循环。

Step 5

Backtest and validate improvements

这一步是整个闭环中的"裁判"——它决定 Step 4 的改动到底有没有用，用的是同一套量化标准、同一批测试数据，容不得半点含糊。

为什么回测和 Step 4 的沙箱不同

Step 4 的沙箱只跑了 3 个测试用例——那是快速冒烟测试，目的是确保改动没有让容器崩溃。Step 5 的回测则是全量对比测试：用 Step 2 建立的完整测试集（历史回放 + 合成边界 + 黄金样本），对修改后的 v(N+1) 配置重新执行一遍完整的三层评分（结构 + 语义 + 稳定性），然后逐指标和 v(N) 的基线比较 delta 值。

三道验证门

第一道：无退化门（No Regression）——检查 v(N+1) 的每一项指标是否都不低于 v(N)。即使一个指标提升了 20%，但另一个指标下降了 2%，也会触发告警。这道门防止的是"拆东墙补西墙"——比如修复了字段缺失问题，但 prompt 变化导致内容质量下降。允许一个很小的容差（默认 2%），避免统计噪声导致误报。

第二道：目标达成门（Target Met）——检查触发本轮优化的那个核心指标是否达到了目标值。如果 Step 3 的提案承诺把 field_complete 从 76% 提升到 95%，但实际只到了 85%，这道门会判定"目标未达成"。这不一定触发回滚——85% 毕竟比 76% 好——但系统会标记这轮优化为"部分成功"，并建议再迭代一轮。

第三道：下游安全门（Downstream Safe）——这是最容易被忽略、但最重要的一道检查。改了 step_02 的输出格式后，step_03 还能正常消费它吗？系统会用 v(N+1) 的实际输出作为输入，跑一遍下游步骤的 Guardian 检查。如果下游的结构校验通过率下降了，说明改动破坏了步骤间的合约——即使 step_02 自身的分数提升了，整体 pipeline 可能变差了。

Delta 计算的细节

回测不是简单地比两个数字。系统对每个指标计算三个维度的 delta：

绝对值 delta——v(N+1) 得分减去 v(N) 得分。这是最直观的：schema pass 从 82% 升到 96%，绝对 delta 是 +14%。

统计显著性——因为 LLM 输出有固有随机性，一个 +2% 的提升可能只是噪声。系统用多次运行（每个测试用例跑 3 次 × 全部测试用例）的数据做简单的 t 检验或 bootstrap 置信区间。只有 delta 在 95% 置信水平上显著为正，才算真正的改进。对于你的 pipeline 规模（10-20 个测试用例 × 3 次运行 = 30-60 个数据点），这已经足够做出可靠判断。

副作用扫描——即使被修改的步骤本身所有指标都提升了，系统还会检查"未被修改的步骤是否受到了影响"。因为步骤是串联的，改了 step_02 的输出格式可能会导致 step_03 的输入解析方式变化。这就是第三道门（下游安全门）做的事。

三种回测结果及其后续动作

全通过（3/3 门）——改动被确认为成功。v(N+1) 成为新的生产版本，系统进入 Step 6 输出优化后的工作流。同时，这轮成功的改动记录（哪种根因 → 哪种修复 → 多少改进）会存入 eval_store，作为未来提案引擎的正反馈——下次遇到类似问题，系统知道这种修复策略是有效的。

部分通过（1-2/3 门）——就像上面的例子。改动确实带来了改善，但没有完全达标，或者有某个指标出现了轻微副作用。系统给你两个选择：接受当前改动并安排下一轮迭代（从 Step 2 重新量化，这次的基线是 v7 而不是 v6）；或者回滚到 v(N) 重新来过。

全失败——改动让情况变更糟了。系统自动回滚到 v(N)，标记这个提案为失败，并记录失败原因。下一轮提案引擎会避免生成类似的修复方案。

防止无限循环

一个关键的工程问题是：如果系统反复迭代，每次都只是"部分通过"，永远达不到目标怎么办？Pipeline Guardian 设置了两个熔断机制。

第一个是最大迭代次数（默认 3 轮）。超过 3 轮仍未达标，系统停止自动迭代，发送一条总结报告给你，列出每轮尝试的修复方案和结果，建议人工介入。

第二个是递减改善阈值。如果第一轮改善了 +14%，第二轮只改善了 +2%，第三轮 +0.5%——系统检测到改善速度在收敛，判断已经接近当前策略的天花板，同样会停止迭代并报告"当前优化空间已基本耗尽，剩余偏差可能需要架构级调整（如拆分步骤或更换模型）"。

这两个机制确保系统不会陷入"改了又改"的死循环，同时也能诚实地告诉你：有些问题不是 prompt 和配置能解决的。

Step 6

Output the optimized workflow

这是闭环的终点，也是下一个循环的起点——系统把经过验证的改进打包成一个可以直接部署的新版工作流。

部署之后：Guardian 不下班

部署不是结束，而是进入持续监控模式。Guardian 节点继续做三件事：

  • 实时评分——每次 cron 触发 pipeline 运行时，Guardian 都会对每个步骤的输出做三层检查并写入 eval_store。
  • 漂移检测——系统每天对比最近 7 天的评分分布与部署时的基线。如果某个指标连续 3 天低于基线超过 5%，触发告警。
  • 自动触发新循环——一旦检测到显著漂移，系统自动从 Step 1 重新开始。整个 6 步不是一次性流程，而是一个永远在转的飞轮。

变更日志：不只记录"改了什么"，更记录"为什么改"

每条变更记录都带有一个 root cause 字段——不是写给机器看的标签，而是写给你（或未来的你）看的一句话解释。三个月后你回看这个 pipeline 的历史，不需要去翻 eval_store 里的原始数据，直接在 changelog 里就能看到："当时 24% 的运行缺少字段，因为 prompt 没有指定输出格式，所以加了 JSON schema 约束"。

这也是给提案引擎的长期记忆。未来处理另一条 pipeline 的类似问题时，系统可以检索历史 changelog 里的成功修复模式："上次 market-intel 的字段缺失问题通过添加 JSON schema 约束解决了，类似策略可能适用于当前情况"。

质量证书：用数据证明改进

质量证书是整个优化周期的"成绩单"，包含三项不可伪造的证据：改动前后的每项指标对比、测试的样本量（20 个输入 × 3 次运行 = 60 个数据点）、以及零退化声明。这份证书的价值在于——如果你将来要向团队成员、合作伙伴或自己证明"这个系统真的有用"，数据全在这里。

部署：怎么生效

部署动作本身极其简单。系统把新版配置文件写入生产路径，更新 YAML 里的文件引用（比如把 prompt_file: step_02_v6.txt 改为 step_02_v7.txt）。不需要重启 Docker 容器、不需要重新构建镜像——因为你的 agent 容器在每次 cron 触发时都会读取最新的配置文件。这意味着今晚改完，明天凌晨的 cron 任务就自动使用新配置运行。

部署完成后，系统通过 Telegram 发送一条简洁的通知：

Changes: prompt rewrite, temperature 0.7→0.3, language constraint added
Composite score: 72% → 89% (+17%)
Rollback: guardian rollback --pipeline market-intel --to v6
Quality timeline — market-intel pipeline

为什么这个设计是"通用的"

回到最初强调的要求——这不是只能改进某两个 pipeline 的工具。整个 Step 6 的输出机制对 pipeline 内容零假设：

配置文件是从 YAML 定义中读取的，不关心你的 agent 具体做什么。变更日志的模板是通用的，任何 pipeline 的任何步骤改动都能以同样的结构记录。质量证书从 eval_store 的通用 eval_traces 表中聚合数据，不绑定特定指标名。回滚命令只操作版本号和文件路径，不涉及业务逻辑。

这意味着当你明天给一条新的 quant-research pipeline 写了 YAML 配置并定义好每个步骤的 schema，它就自动获得了和 market-intel 完全相同的优化-回测-发布能力，不需要改一行 Guardian 代码。

六步走完，我们回头看整个系统做了什么：从一条"盲跑"的 pipeline，变成了一条每个接缝都有质量闸门、每次运行都有量化记录、出了问题能自动诊断和修复、并且持续监控漂移的自愈系统。整个过程中你作为人类的介入点只有两个：批准 Step 3 的提案、以及在 Step 5 部分通过时决定"接受还是回滚"。其余的分析、量化、修复、回测、部署、监控全部自动完成。
